# Avocado RUL Labeling UI and Inspection Workflow

## Purpose

This document describes the avocado RUL labeling interface and the related inspection workflow. It focuses only on the tools used to construct and review End-of-Life (EOL), Remaining Useful Life (RUL), and failed-frame exclusion metadata for the avocado dataset.

The UI was created after avocado segmentation was completed. Its purpose is to support human-confirmed label construction before the model-ready dataset and sequence windows are built.

## Labeling Context

For the current MVP, EOL is defined as the first visually confirmed point where external spoilage or visible mold onset indicates that the fruit is no longer acceptable for the intended edible-status task.

This is not an internal quality guarantee. Internal fruit condition would require destructive testing or additional internal-quality measurements. Firmness, environmental readings, RGB/LAB trends, and visual darkening are used as supporting evidence during manual review.

RUL is computed as:

```text
RUL hours = EOL timestamp - frame timestamp
```

Negative RUL values are clipped to zero in the inspection output.

## UI Components

The UI supports:

- fruit selection by `fruit_id`;
- day filtering with `All days` or a specific date;
- frame slider and previous/next navigation;
- play, repeat, shuffle fruit, and start-from-EOL playback controls;
- adjustable playback delay in milliseconds;
- adjustable frame skip during playback;
- segmented avocado image display;
- environment graph and numeric readout;
- firmness graph and numeric readout;
- RGB and LAB temporal graphs;
- saved EOL marker on graphs;
- active and saved cut-range markers on graphs;
- notation panel for short reminders such as EOL, CUT, RUL, and current fruit identity.

Playback starts from the beginning of the selected fruit/day timeline by default. When `Start from EOL` is enabled, playback starts from the saved EOL point for the selected or shuffled fruit.

## EOL Annotation Workflow

1. Open the UI:

```bash
python src/avocado/stage3_preprocessing/eol_labeling_ui.py
```

2. Select a fruit, such as `F01`.
3. Optionally select a specific day.
4. Review the segmented fruit images with the slider or playback controls.
5. Use firmness, environment, RGB, LAB, and visual progression as supporting evidence.
6. Select the EOL basis and confidence.
7. Add a visual note if needed.
8. Click `Mark Current Frame As EOL`.
9. Click `Save All Annotations`.

The saved EOL marker appears on the graphs so the selected point can be reviewed visually against the supporting signals.

## Failed-Frame Exclusion Workflow

The UI can also record failed frames as non-destructive cut ranges.

1. Move to the first failed frame.
2. Click `Set Start`.
3. Move to the last failed frame.
4. Click `Set End`.
5. Choose a reason, such as `bad_segmentation`, `blur`, `wrong_crop`, or `lighting_artifact`.
6. Add a note if needed.
7. Click `Add Cut Range`.
8. Click `Save Cut Ranges` or `Save All Annotations`.

The UI does not delete or modify image files. It only writes exclusion metadata. Downstream dataset construction should read `frame_exclusions.csv` and apply the exclusions reproducibly.

## EDA and RUL Inspection Workflow

The notebook `notebooks/avocado_rul_dataset_inspection.ipynb` is a wrapper. It does not contain the primary processing logic. Instead, it calls the Python scripts and previews the generated tables.

The intended order is:

1. Run EDA:

```bash
python src/avocado/stage2_eda/run_avocado_eda.py
```

2. Run RUL inspection:

```bash
python src/avocado/stage3_preprocessing/inspect_rul_dataset.py
```

3. Review generated tables in:

```text
data/02_processed/avocado/rul_inspection/
```

Both scripts skip work when their expected outputs already exist. Use `--force` when regeneration is needed.

## Design Decisions

- The UI is human-assisted because visible spoilage and mold onset require visual confirmation.
- EOL labels are fruit-level anchors, not frame-level independent labels.
- Failed-frame cuts are saved as metadata, not applied destructively.
- EDA features support review and inspection but should not be treated as the primary label source.
- Model-ready sequence construction should happen downstream after reviewing `eol_annotations.csv`, `frame_exclusions.csv`, and the RUL inspection outputs.

## Known Limitations

- The EOL decision is based on visible external condition.
- Internal avocado quality is not measured.
- Firmness is daily, while image/environment records are more frequent.
- Exclusion ranges must be applied by downstream data-construction scripts.
- Current outputs prepare RUL inspection and baseline-ready frame filtering, but they are not yet final sequence windows.
